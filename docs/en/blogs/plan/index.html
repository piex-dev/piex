Plan Mode: Think Before You Edit

Institutionalize research-before-edits.

💡

In short

/plan isn’t another command—it’s a regime: write tools locked, plans reviewable, progress visible.

Background

Agents fail by editing too early, scope creep, invisible progress, and lost plans after compaction. Plan Mode is workflow, not another generator.

@piex-dev/plan toggles read-only planning, tracks todos, and shows footer progress.

Strip write tools for real; bash goes through a shell-lexer allowlist (quote/escape/segment parsing; redirection, subshells and expansion rejected; read-only git subcommands with anti-textconv argument guards). The agent asks clarifying questions via plan_question and submits the plan via plan_complete (the Plan:-heading regex is now a fallback). Mode instructions are appended to the systemPrompt by before_agent_start — zero message-history pollution. Advance via [DONE:n], persist with appendEntry + PLAN.md.

Implementation

~1430-line single file exercising the full Extension API surface: registerCommand, registerTool (plan_question/plan_complete), setActiveTools, tool_call blocking, before_agent_start → systemPrompt, turn_end parsing, status/widget UI. Lighter than omp's overlay, intentionally.

Design notes

Project Mechanism piex choices
pi official plan-mode example registerCommand + setActiveTools + tool_call bash interception Adopted core API approach: tool switching, bash interception, DONE marker parsing. Enhanced: appendEntry cross-turn persistence + PLAN.md file + UI widget
oh-my-pi plan Full-screen plan-review overlay + TOC sidebar + sub-agent handoff Dropped: pi API lacks full-screen overlay support, sub-agent not ready. Borrowed: step approval (replaced with select), configurable tool whitelist, review integration
pi-extensions pi-plan-mode Shell-lexer bash allowlist, plan_mode_question/plan_mode_complete structured tools, systemPrompt injection, context artifact cleanup Adopted all four (replacing the regex blacklist, prose regex parsing and message injection); dropped: settings-file migration, fixed thinking level, and its lack of execution tracking (piex keeps DONE tracking)

Key tradeoff: stay close to pi example (light, predictable, maintainable), drop omp heavy UI; use appendEntry for compaction resistance.

Version history

Version Bash safety Plan / questions Directive injection Notes
0.1.1 regex blacklist prose Plan: heading regex context message Blocks known-bad commands; plan extracted from Plan: heading regex in prose; constraint messages injected into the message stream
0.2.0 (current) shell-lexer allowlist plan_complete/plan_question structured tools systemPrompt Default-deny, rule-based allowlist; plan and questions via dedicated tool params; directives rebuilt as system prompt each turn, leaving no plan artifacts in message history

The 0.1.1 lesson: a regex blacklist can't stop equivalent bypasses (find -delete, $(), ; chaining), prose regex extraction misses plans when the model deviates from format, and context-injected constraints get swallowed by compaction and themselves become artifacts to clean up. 0.2.0 swaps all three to structured alternatives at once: allowlist shrinks the bypass surface, tool params replace prose parsing, systemPrompt replaces message injection, and the whole class of boundary problems disappears.

Roadmap

Weak DONE checks (flag DONE-without-changes), richer refine approval, configurable bash allowlist with "risk reduction ≠ isolation" documented, /review handoff, PLAN.md frontmatter.
