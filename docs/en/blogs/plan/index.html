Plan Mode: Think Before You Edit

Institutionalize research-before-edits.

💡

In short

/plan isn’t another command—it’s a regime: write tools locked, plans reviewable, progress visible.

Background

Agents fail by editing too early, scope creep, invisible progress, and lost plans after compaction. Plan Mode is workflow, not another generator.

@piex-dev/plan toggles read-only planning, tracks todos, and shows footer progress.

How it works

Strip write tools for real, gate destructive bash, parse numbered plans, advance via [DONE:n], persist with appendEntry + PLAN.md.

Implementation

One extension file exercising the full Extension API surface. Lighter than omp’s overlay, intentionally.

Design notes

ProjectMechanismpiex choices
pi official plan-mode exampleregisterCommand + setActiveTools + tool_call bash interceptionAdopted core API approach: tool switching, bash interception, DONE marker parsing. Enhanced: appendEntry cross-turn persistence + PLAN.md file + UI widget
oh-my-pi planFull-screen plan-review overlay + TOC sidebar + sub-agent handoffDropped: pi API lacks full-screen overlay support, sub-agent not ready. Borrowed: step approval (replaced with select), configurable tool whitelist, review integration

Key tradeoff: stay close to pi example (light, predictable, maintainable), drop omp heavy UI; use appendEntry for compaction resistance.

Roadmap

Sturdier plan IR, weak DONE checks, review handoff, PLAN.md frontmatter, configurable allowlists. Not a sandbox.
