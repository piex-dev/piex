Hashline — Principles & Lessons

Line numbers are fragile coordinates. Anchored editing replaces them with content hashes so agents can edit safely under concurrency. This post compares three designs and explains piex/hashline’s choices.

💡

30-second takeaway

piex inherits oh-my-pi’s tree-sitter block ops and boundary repair, then fills in Noop Loop / Duplicate Edit / dialect normalization from pi-hashline-edit. Hash model stays whole-file 4-hex tags—simple and verifiable; Phase 2 adds stale recovery and multi-version snapshots.

Why anchored editing

AI coding assistants face a fundamental edit problem: models describe locations by line number, but line numbers are fragile. The user may edit the file while the model thinks; the model’s own previous edit shifts every later line. Trusting bare line numbers puts edits in the wrong place—bugs that are hard to notice.

Anchored editing (Hashline) replaces line numbers with content hashes as the coordinate system. On read, each line is shown with a hash; on edit, the model points at hashes (“replace between this hash and that hash”) instead of “lines 12–15.”

piex/hashline draws on three strong hashline designs. This post compares them—decisions, strengths, trade-offs—and explains piex/hashline’s choices.

2. Whole-file hash vs per-line hash

This is the fundamental design fork.

2.1 Whole-file hash (oh-my-pi)

text
On read:
  full file → xxHash32 → top 16 bits → 4-hex tag "A3F2"
  output: [src/main.ts#A3F2]
          1:A3F2:import { foo } from "./foo";
          2:B7E1:const x = 1;

On edit:
  [src/main.ts#A3F2]        ← tag proves "I saw this version"
  SWAP 2.=2:                ← line + tag dual anchor
  +const x = 2;

Validation:

  1. Re-read file → hash → actualTag
  2. Compare to header expectedTag
  3. Match → line indices refer to the tagged version; safe to apply
  4. Mismatch → MismatchError, re-read required

Pros: simple mental model; one tag covers the file; seenLines can block edits to unread lines.

Cons: any change invalidates the global tag; very sensitive to concurrent edits; model must re-read the whole file to edit anywhere.

2.2 Per-line hash (pi-hashline-edit, pi-hashline-edit-pro)

text
On read:
  hash each line independently
  output: 1#MQ:import { foo } from "./foo";
          2#VR:const x = 1;

On edit:
  { "op": "replace", "pos": "2#VR", "lines": ["const x = 2;"] }
  ← anchored on line 2 hash VR

Validation:

  1. Hash the line at the anchor’s line number in the live file
  2. Compare to expectedHash
  3. Match → line unchanged; safe
  4. Mismatch → E_STALE_ANCHOR

Pros: local invalidation—edit line 5, line 50 still holds; read only the needed range.

Cons: collisions matter more (same content lines need extra policy).

2.3 Context-aware hashing (pi-hashline-edit only)

text
computeHashFromContext(prev, curr, next) =
  xxHash32(prev + "\0" + curr + "\0" + next)

Line N’s hash depends on N−1, N, N+1. Edit radius is exactly ±1:

text
edit line 5 →
  line 4 stale (next changed)
  line 5 stale (self)
  line 6 stale (prev changed)
  lines 3, 7, 8… still valid

Looser than whole-file (global fail), safer than pure content hash (only the edited line)—neighbors of the edit should not be trusted.

2.4 Stable hash mapping (pi-hashline-edit-pro only)

Pure content hashes, plus stable hash mapping after edits: match old hashes to same-content lines in the new file so unchanged lines keep old hashes. The model can keep editing the same file without re-read after a successful edit.

Invalidation ranges

text
whole-file (oh-my-pi):

context-aware (pi-hashline-edit):
  edit line 5 → ___███______________  ±1

content + stable mapping (pi-hashline-edit-pro):
  edit line 5 → ____█_______________  only the edited line; rest survive

3. Tolerance mechanisms

3.1 Stale-anchor recovery

oh-my-pi — Recovery (anchor remapping)

text
1. diffArrays(old, new) → Map<oldLine, newLine>
2. verify non-anchor neighbor continuity after remap
3. all anchors must share one offset
4. pass → remap and apply on new file
5. fail → MismatchError

Strict and conservative: one bad anchor rejects all.

pi-hashline-edit — 3-way merge

text
1. take last model read snapshot (LRU: up to 4 versions × 8 paths)
2. replay edit on snapshot → baseEdited
3. structuredPatch(snapshot, baseEdited)
4. applyPatch(live, patch, {fuzzFactor: 0})
5. success → merged result + warning
6. fail → older snapshots; else E_STALE_ANCHOR

fuzzFactor=0 is the key: conflicting edits on the same line fail the merge—no silent overwrite.

pi-hashline-edit-pro — no recovery; relies on stable mapping to lower failure rate.

3.2 Noop loop guard (pi-hashline-edit only)

Models often ignore soft hints and retry the same noop payload. Extreme case: 205 byte-identical noop edits in one session.

text
same (canonicalPath, payloadKey) → noopCount++
noopCount ≥ 3 → ToolError "[E_NOOP_LOOP]"
new payload → reset counter

oh-my-pi and pi-hashline-edit-pro lack this.

3.3 Duplicate-edit detection (pi-hashline-edit only)

Orthogonal to noop guard: after a successful edit the model may resend the same payload (e.g. append once, then again → duplicated content).

text
lastAppliedPayload[path] = payloadKey
new request matches && read snapshot == live → [E_DUPLICATE_EDIT]
model re-reads → clear lastAppliedPayload

3.4 Boundary repair

oh-my-pi’s repair engine (~1200 LOC) is the most complete: boundary echo, delimiter imbalance, JSX closer mismatch, after-insert landing by indent. The per-line designs only have simplified versions (warn or autoFix trim).

3.5 Dialect normalization (pi-hashline-edit only)

Models call edit with non-schema shapes—Pi native oldText/newText, stringified edits JSON, file_path instead of path. prepareArguments normalizes before validation so a bad shape still runs without an extra model turn.

3.6 textHint cross-check (pi-hashline-edit only)

Anchor form LINE#HASH:content may carry optional textHint:

  • hash matches but textHint does not → stale (guards 1/256 collision)
  • hash misses but textHint matches fuzzily → accept (whitespace tolerance)

textHint is free—already present in read output.

4. Edit operation models

4.1 Three syntax styles

oh-my-pi — custom DSL

text
[src/main.ts#A3F2]
SWAP 12.=12:
+const x = 1;

DEL 5
INS.PRE 8:
+import { foo } from "./foo";

SWAP.BLK 20:             ← whole syntactic block via tree-sitter
+function greet(name) {
+  return `Hello, ${name}`;
+}

REM
MV lib/greet.ts

pi-hashline-edit — JSON multi-op

json
{
  "path": "src/main.ts",
  "edits": [
    { "op": "replace", "pos": "12#MQ", "lines": ["const x = 1;"] },
    { "op": "append", "pos": "8#VR", "lines": ["import { foo }..."] },
    { "op": "replace_text", "oldText": "var x = 1", "newText": "const x = 1" }
  ]
}

pi-hashline-edit-pro — JSON single op

json
{
  "path": "src/main.ts",
  "changes": [
    { "content_lines": ["const x = 1;"], "hash_range_inclusive": ["MQX", "MQX"] }
  ]
}

4.2 Block ops — oh-my-pi’s unique edge

SWAP.BLK N: uses tree-sitter to resolve the full syntactic block starting at line N. Whether the function is 3 or 30 lines, the range is exact. Only design with syntax awareness.

4.3 Op diversity — pi-hashline-edit’s edge

OpNotes
replace + possingle line
replace + pos + endrange
append / prependafter/before line or EOF/BOF
replace_textexact string replace

oh-my-pi’s DSL is more expressive (blocks + file ops) but harder for models to learn. pi-hashline-edit-pro is replace-only.

6. piex/hashline’s choice

Current architecture: oh-my-pi as engine

piex/hashline uses oh-my-pi as the core engine, with a Node.js adapter. That keeps:

  • tree-sitter block ops (SWAP.BLK / DEL.BLK / INS.BLK.POST)
  • boundary repair (~1200 LOC)

Roadmap

text
Phase 1 ✅ done — high ROI (~180 LOC):
  ├── Noop Loop Guard
  ├── Duplicate Edit detection
  └── Dialect normalization

Phase 2 — reliability (~550 LOC):
  ├── Stale Anchor recovery (3-way merge)
  ├── Multi-version snapshot store (LRU)
  └── Grep with anchors

Phase 3 — polish:
  ├── Undo / Auto-Read / Raw Read / Stable Hash Mapping

Phase 1 lives in extensions/patches.ts (EditGuard) and normalizeInput() in extensions/hashline.ts.
