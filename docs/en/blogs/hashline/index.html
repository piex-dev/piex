2026-07-14 Hashline Edit Extension

Hashline: Content Anchors, Not Fragile Line Numbers

Default edits shift from guessing lines to editing the version you actually read.

💡

In short

With @piex-dev/hashline, edits are version-checked: fewer silent wrong writes, occasional re-reads. A tax worth paying.

Background

Agent edit tools trust line numbers, but lines drift: users edit mid-thought, prior patches shift offsets, models invent lines they never read. Silent wrong writes are worse than hard failures.

@piex-dev/hashline replaces pi’s built-in edit with content-anchored patches: tag on read, verify on write, refuse unseen lines.

How it works

Reads inject [path#TAG] and record seen-lines. Edits must carry the same whole-file 4-hex tag (optimistic lock). Stricter than per-line hashes, but unlocks tree-sitter block ops and boundary repair from oh-my-pi.

hashline
[src/main.ts#A3F2]
SWAP 2.=2:
+const x = 2;

Phase 1 guards: three identical no-ops → [E_NOOP_LOOP]; resending a successful payload on an unchanged file → [E_DUPLICATE_EDIT]; normalize CRLF / fenced DSL.

Implementation

Thin wrapper over @oh-my-pi/hashline: override edit, hook read, Node FS + realpath keys, EditGuard, and a Bun xxHash32 polyfill.

CapabilityStatus
tree-sitter blocks / REM·MV✅ from engine
Noop / Duplicate / normalize✅ Phase 1
Stale recovery / multi-version snaps❌ Phase 2

Design notes

ProjectMechanismpiex choices
oh-my-pi hashlineFull-file tag + tree-sitter block syntax + boundary repair engineAdopted: wrapping @oh-my-pi/hashline, inheriting tags, SWAP.BLK, REM/MV and engine. Dropped: built-in integration, Bun FS, LSP writethrough; replaced with Node + outer EditGuard
pi-hashline-editPer-line context hashing + 3-way merge recovery + JSON DSLDropped core algorithm (divergent route), borrowed fault-tolerance intent (noop/dup guard). ADRs kept as Phase 2 stale recovery references
pi-hashline-edit-proPer-line content hashing + stable mappingDropped algorithm, borrowed "minimize re-read" as long-term target

Key tradeoff: whole-file tag (strict & simple) over per-line (local resilience), fault-tolerance in wrapping layer. Three phases: Phase 1 guards (done) → Phase 2 recovery → Phase 3 undo/LSP integration.

Roadmap

Prefer merge over pure reject when safe; multi-version snapshots + anchored grep; finer duplicate keys; wrapper tests and optional LSP refresh.

Appendix

Three community lines: whole-file tags, context per-line hashes, stable mapping. piex keeps the first as engine and borrows tolerance ideas. Full matrix in the source note.
