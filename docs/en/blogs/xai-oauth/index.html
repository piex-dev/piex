xAI OAuth: Subscription Login for Grok

Use subscription quota; keep keys for automation.

💡

In short

Already paying for Grok? @piex-dev/xai-oauth uses that quota. API keys remain for metered automation.

Background

Built-in xAI is API-key metered. Subscribers already pay SuperGrok / X Premium+ and want that quota, plus live model catalogs.

@piex-dev/xai-oauth adds device-code OAuth beside the key provider.

How it works

RFC 8628 device flow, dual catalog discovery (public API + subscription proxy), reload to pick up new models.

Implementation

OAuth + models modules, safe error formatting, endpoint checks, env allowlists, unit tests without the network.

Design notes

ProjectMechanismpiex choices
oh-my-pi xai-oauthDevice Authorization Grant full flow; dual-endpoint model discovery; pi standard OAuth provider registrationAdopted: OAuth flow chain, client_id, endpoint validation, error truncation and skew. Dropped: Bun runtime details (switched to Node)
stnly/pi-grokModel discovery: fetch /v1/models + merge into provider tableBorrowed: live discovery concept (/reload for new models), but discovery and routing reimplemented independently
pi built-in xaiAPI Key auth, fixed model setCoexisting: same instance can use built-in xai (Key/CI) + xai-oauth (subscription/interactive)

Key tradeoff: subscription tokens in pi standard storage (extension layer never persists), best-effort model discovery (fallback on failure), safety: "err on the side of silence".

Roadmap

Observable discovery sources, clearer quota/auth errors, stronger provider labeling, contract tests.
