2026-07-19 LSP Diagnostics Extension

LSP：给 Agent 一双「语言服务器」的眼睛

更关键的是 edit 之后自动看到 ERROR，不必靠模型记得再调诊断。

💡

导语

@piex-dev/lsp 让 agent 能问 IDE 同一类问题；更关键的是 edit 之后自动看到 ERROR，不必靠模型记得再调诊断。

问题背景

没有 LSP 时，agent 只靠 read/grep 与脑补类型。改签名漏调用点、红线要等整次构建才知道。人编辑器交给语言服务器；agent 也该能问。

@piex-dev/lsp：导航、诊断、rename/code_actions，以及写后 ERROR 闭环。

技术原理

stdio JSON-RPC；rootMarkers 发现 server；publishDiagnostics 缓存。写路径 hook：sync（didOpen/didChange）→ 等诊断 → 仅 ERROR 裁剪后附在 tool 结果。

实现方案

兼容 initOptions/settings；多 server 聚合诊断、linter 不抢导航；rename 默认 preview；WorkspaceEdit 限 cwd。PI_LSP_DIAGNOSTICS_ON_EDIT=0 可关写后诊断，PI_<NAME>_LSP_COMMAND 可覆盖单 server 命令。

诊断正确性：push 诊断等静默期（默认 800ms，diagnosticsSettleMs 可按 server 配）再返回，慢 server 先推空后推真不会误报干净；server 声明 diagnosticProvider 时改用 LSP 3.17 pull 诊断。codeAction/resolve 只在声明 resolveProvider 时调用；重叠 TextEdit 拒绝应用；server stderr 捕获进错误消息；Windows .bat/.cmd 经 cmd.exe 包装。

闭环：hashline edit → 自动 ERROR → 需要时显式 lsp 导航/重构 → 运行时问题上 dap。

设计参考

项目机制piex 取舍
oh-my-pi lsp完整 LSP 客户端：多 server 路由、didChange、完整 action 面、诊断聚合采纳：JSON-RPC client、defaults.json 驱动、按需启动/会话复用、多 server 聚合。不采纳：Bun 运行时（改 Node）、大面铺满 action
OpenCode 写后诊断tool_result hook 拦 edit/write → 等 publishDiagnostics → 仅 ERROR、每文件 cap采纳整套模式：sync → wait → only ERROR → cap 20 → 干净文件不附加
VS Code LSPinitialize + settings/didChangeConfiguration + workspace/configuration借鉴：initOptions/settings 正确下发路径；full-text didChange；which 查 node_modules/.bin 和 .venv
pi-extensions pi-lsp诊断 settle 静默期、LSP 3.17 pull 诊断、stderr 捕获、resolveProvider 门控、重叠 edit 检测、cmd.exe 包装、命令 env 覆盖采纳全部协议细节（融入常驻进程架构）；不采纳：spawn-per-call、仅 diagnostics/fix 两工具（piex 保留 13 action）

核心取舍：优先写后 ERROR 闭环（学 OpenCode），诊断优于导航暴露；linter 不抢 primary server 的导航角色。

版本迭代

版本诊断时序能力门控错误现场进程模型
0.2.0push 诊断到即返，慢服务器先推空会被误报「干净」盲调 codeAction/resolveserver 退出只给 exit code，stderr 丢失会话内复用，Windows .bat 直接 spawn 失败
0.3.0（当前）push settle 静默期 + LSP 3.17 pull 双轨resolveProvider/diagnosticProvider 声明才调stderr 捕获进超时/退出错误，排障不用猜会话内复用不变，.bat/.cmd 经 cmd.exe 包装，PI_<NAME>_LSP_COMMAND 覆盖

0.2.0 的教训：intelephense 这类 server 会先推一批空诊断、再推真诊断，到即返会把有错的文件报成干净；server 崩溃时只有 exit code，排障全靠猜。0.3.0 把协议细节补齐——settle 静默期等推送稳定、pull 诊断让 server 按需算、stderr 进错误消息、resolveProvider 门控避免对不支持的 server 发多余请求、重叠 TextEdit 检测防写坏文件。这些都是 pi-extensions pi-lsp 先跑通的协议经验，piex 融进常驻进程架构，冷启动只付一次。

优化计划

已完成正确性、写后诊断、rename/code_actions、诊断 settle/pull、重叠 edit 防护。单测：bun test packages/lsp。下一步：项目级 .lsp.json、模块拆分、indexing ready、目录级批量诊断。暂缓 lspmux/自动下载/completion。
