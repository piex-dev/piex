Hashline 方案的原理及借鉴

行号是脆弱的坐标。锚定编辑用内容哈希替代行号，让 agent 在并发修改下仍能安全 edit。 本文对比三种实现，说明 piex/hashline 为何以 oh-my-pi 为引擎，又补了哪些 Phase 1 容错。

💡

30 秒结论

piex 继承 oh-my-pi 的 tree-sitter block 操作 与 boundary repair， 再用 pi-hashline-edit 思路补齐 Noop Loop / Duplicate Edit / 方言归一化。 哈希模型仍是全文件 4-hex tag——简单、可验证；Phase 2 再补 stale 恢复与多版本快照。

为什么需要锚定编辑

AI 编码助手的 edit 工具面临一个根本问题：模型用行号描述编辑位置，但行号是脆弱的。 用户可能在模型思考期间修改文件，模型自己前一轮的编辑也会让后续行号全部偏移。 直接信任行号，edit 会在错误位置修改代码，造成难以察觉的 bug。

锚定编辑（Hashline）的思路是：用内容哈希代替行号作为坐标系统。 模型读取文件时看到每行旁边的哈希；编辑时用哈希告诉工具「替换从这里到那里」，而不是「替换第 12 到 15 行」。

piex/hashline 从三个优秀实现中汲取灵感。下面按设计分歧展开。

三个参考实现

oh-my-pi

全文件 4-hex tag

一个 tag 证明「我看到的是这个版本」。概念最简单，含 tree-sitter block 与完整 boundary repair。

~3500 行 12 tests 语法感知
pi-hashline-edit

上下文感知逐行哈希

哈希依赖 prev/curr/next。局部失效 + 3-way merge 恢复 + 丰富容错，与 oh-my-pi 无共享代码。

~4000 行 30+ tests 容错最全
pi-hashline-edit-pro

纯内容哈希 + stable map

编辑后按内容重映射哈希，未改行锚点存活。支持 Undo / Auto-Read / Flat Mode。

~3500 行 50+ tests 少 re-read
实现 哈希粒度 作者
oh-my-pi/hashline 全文件 → 4-hex tag can1357
pi-hashline-edit 逐行上下文感知 → 2/3/4 char RimuruW
pi-hashline-edit-pro 逐行内容 → 3 char YuGiMob
ℹ️

关键事实

后两者与 oh-my-pi 没有任何共享代码。 pi-hashline-edit 注释写 “Vendored & adapted from oh-my-pi”，但核心算法（上下文哈希、textHint、3-way merge）全部是原创。

原理：整体哈希 vs 逐行哈希

这是最根本的设计分歧。

整体哈希（oh-my-pi）

read → edit flow 
读取时：
  全文件内容 → xxHash32 → 截取前 16 bit → 4-hex tag "A3F2"
  输出：[src/main.ts#A3F2]
          1:A3F2:import { foo } from "./foo";
          2:B7E1:const x = 1;

编辑时：
  [src/main.ts#A3F2]        ← 用 tag 证明"我看到的是这个版本"
  SWAP 2.=2:                ← 行号 + tag 双重定位
  +const x = 2;
1

校验流程

重读 → 算 actualTag → 与 header expectedTag 对比 → 匹配才 apply，否则 MismatchError。

+

优点

概念简单；seenLines 可阻止编辑从未读取的行。

−

缺点

任意一处改动 → 全局 tag 失效；必须先读完整文件才能编辑任意位置。

逐行哈希（后两者）

per-line anchor 
读取时：
  1#MQ:import { foo } from "./foo";
  2#VR:const x = 1;

编辑时：
  { "op": "replace", "pos": "2#VR", "lines": ["const x = 2;"] }
  ← 锚定在行 2 的哈希 VR 上

优点：局部失效——只改第 5 行，第 50 行锚点仍有效，减少 re-read round-trip。 缺点：哈希碰撞更突出，需要额外策略。

上下文感知哈希（pi-hashline-edit）

computeHashFromContext 
computeHashFromContext(prev, curr, next) =
  xxHash32(prev + "\0" + curr + "\0" + next)

第 N 行哈希依赖 N−1、N、N+1。编辑第 5 行只让 4/5/6 失效，远处锚点保持有效—— 比整体哈希宽松，比纯内容哈希更安全。

Stable Hash Mapping（pi-hashline-edit-pro）

纯内容哈希 + 编辑后按内容匹配把旧哈希映射到新文件相同行。 模型一次编辑后不必 re-read 就能继续编辑。

失效范围对比

整体哈希 (oh-my-pi) 任意改动 → 全部失效
上下文感知 (pi-hashline-edit) 编辑第 5 行 → ±1 行
内容 + stable map (pro) 仅编辑行本身失效

容错机制对比

Stale Anchor 恢复

oh-my-pi

Recovery · anchor remapping

diffArrays 建行号映射；所有锚点必须统一 offset。严格但保守——一个对不上，全部拒绝。

pi-hashline-edit

3-way merge

在 read 快照上重放编辑，生成 patch 合并到 live 文件。fuzzFactor=0：冲突必失败，不静默覆盖。

pi-hashline-edit-pro 无恢复，依赖 stable mapping 降低失效概率。

模型行为守卫

⟳

Noop Loop Guard

同一 payload 连续 ≥3 次 byte-identical noop → 抛 [E_NOOP_LOOP]。真实案例：205 次重复 noop。

⎘

Duplicate Edit

成功 edit 后重发相同 payload（以为失败）→ [E_DUPLICATE_EDIT]。re-read 后清除记录。

≡

方言归一化

file_path→path、JSON 字符串 edits、oldText/newText 等变异在验证前吸收，省一轮 turn。

✓

textHint 交叉校验

锚点可选 :content 后缀：哈希匹配但内容不符 → 过期；哈希不符但内容模糊等价 → 可接受。

后三项主要为 pi-hashline-edit 独有；oh-my-pi 的强项在 Boundary Repair 引擎（~1200 行）：自动裁剪 boundary echo、修复括号/JSX closer 不平衡、按缩进调整插入位置。

编辑操作模型

oh-my-pi · DSL

DSL 
[src/main.ts#A3F2]
SWAP 12.=12:
+const x = 1;

DEL 5
INS.PRE 8:
+import { foo } from "./foo";

SWAP.BLK 20:   ← tree-sitter 整块
REM            ← 删文件
MV lib/a.ts    ← 重命名

pi-hashline-edit · JSON

JSON ops 
{
  "path": "src/main.ts",
  "edits": [
    { "op": "replace", "pos": "12#MQ",
      "lines": ["const x = 1;"] },
    { "op": "append", "pos": "8#VR",
      "lines": ["import ..."] },
    { "op": "replace_text",
      "oldText": "var x",
      "newText": "const x" }
  ]
}
◆

Block 操作

SWAP.BLK N: 通过 tree-sitter 解析第 N 行所在的完整语法块。 这是三个实现中唯一具备语法感知的设计——模型不必精确计算函数起止行号。

piex/hashline 的选择

piex/hashline 以 oh-my-pi 为引擎，在其上做 Node.js 适配与容错层。保留最独特的两点：

  • tree-sitter block 操作（SWAP.BLK / DEL.BLK / INS.BLK.POST）
  • boundary repair 引擎（自动修边界错误）
维度 oh-my-pi piex/hashline pi-hashline-edit
引擎 自身 封装 oh-my-pi 独立重写
规模 ~3500 行 ~200 行封装层 ~4000 行
Block 操作 ✅ ✅ 继承 ❌
Boundary Repair ✅ ✅ 继承 ⚠️
Noop / Dup / 方言 ❌ ✅ Phase 1 ✅
Stale 恢复 ✅ ❌ ✅

结论：继承层全 ✅，Phase 1 容错已完成 ✅。 实现见 extensions/patches.ts（EditGuard）与 normalizeInput()。
